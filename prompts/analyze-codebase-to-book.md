# 提示詞：將任何程式碼庫轉化為技術書籍

一套可重複使用的系統，用於分析程式碼庫並透過平行 AI 代理產出出版品質的技術書籍。設計目標是產出可與 O'Reilly 技術書媲美的成品。

---

## 提示詞內容

```
我希望你分析位於 [path] 的原始碼，並產出一本關於其架構、模式和內部機制的
全面性技術書籍。

這本書的閱讀體驗應如同專業技術出版品——那種資深工程師會購買來深入理解一個
系統的書。不是文件。不是教學。而是一本教導系統如何運作、每個決策為何如此做出、
以及讀者可以借鑑哪些模式用於自己專案的書。

---

## 第一階段：探索

啟動平行代理（每個主要子系統一個），exhaustive 地閱讀程式碼庫中的每個檔案。
每個代理應記錄：

- 架構與模組邊界
- 關鍵抽象（型別、介面、核心類別）
- 資料流（資訊如何在系統中流動）
- 設計模式（使用了哪些模式以及為什麼）
- 整合點（此模組如何與其他模組連接）
- 出人意料的決策（任何非顯而易見或巧妙的設計）

為每個子系統產出一份原始分析文件。這些是研究筆記，不是最終書籍。

## 第二階段：受眾與定位

在組織書籍結構之前，先定義：

**主要受眾**：這本書是寫給誰的？他們已經知道什麼？他們想學什麼？
本書應同時服務兩類讀者：
- 技術主管，想了解架構與設計背後的理由（可以跳過程式碼區塊和深入探討段落）
- 資深工程師，想獲得實作層級的理解（閱讀所有內容，包括深入探討）

**核心論點**：關於這個系統，那個唯一的重大洞見是什麼？每一章都應回扣這個
論點。對於一個程式碼庫，這通常是：「這是這個系統所下的架構賭注，而這是每個
子系統如何服務於這個賭注。」

**為什麼值得寫成一本書**：為什麼讀者不能直接讀原始碼就好？本書的價值在於：
敘事（原始碼沒有敘事）、橫切面模式（散落在原始碼的各個檔案中）、設計理由
（完全不在程式碼裡）、以及可轉移的經驗（需要原始碼無法提供的綜合分析）。

## 第三階段：結構

像讀者要從零開始建構這個系統一樣來組織書籍。每一章應解決一個明確的問題，
而下一章依賴於此問題的解答。讀者不應遇到需要後面章節才能理解的概念。

**部分**：將章節分組為 5-7 個主題部分。每個部分有一行題詞來界定該段落。
部分創造自然的閱讀中斷點，並讓目錄更易於瀏覽。

**章節排序原則**：
- 基礎先行（啟動、狀態、與外部服務的通訊）
- 核心迴圈次之（主要執行循環——這通常是最重要的一章）
- 建構於核心之上的能力（工具、外掛、擴展）
- 進階模式（多代理、編排、協調）
- 支援基礎設施（UI、網路、持久化）
- 效能與最佳化放最後（你無法最佳化你不理解的東西）
- 結語：綜合、可轉移的經驗、展望未來

**章節篇幅**：目標每章 300-800 行。若一章超過 800 行，代表它試圖涵蓋太多
——拆分它。若一章不到 200 行，可能不足以成為獨立章節——合併它。

呈現完整大綱，包含部分名稱、章節標題，以及每章 2-3 個描述其涵蓋內容的要點。
在開始撰寫前取得核准。

## 第四階段：撰寫

以第一階段的分析作為研究筆記，從頭開始撰寫每一章。不要重新組織分析——
將其改寫為敘事散文。

### 章節範本

每一章遵循此結構：

1. **開場**（2-3 段）
   - 這個層次/子系統解決什麼問題？
   - 它為何存在？沒有它會出什麼問題？
   - 它如何與讀者已知的內容連結？（明確回溯引用前一章）
   - 讀完後讀者將理解什麼？

2. **正文**（核心內容）
   - 混合散文、圖表、程式碼片段和表格
   - 散文用於敘事和理由（「為什麼」）
   - 圖表用於架構、資料流和狀態機
   - 程式碼用於關鍵模式（僅限虛擬碼——見下方規則）
   - 表格用於參考資料（欄位列表、設定選項）

3. **深入探討段落**（選用，行內呈現）
   - 技術主管可以跳過的實作細節補充段落
   - 包含「這在位元層級上到底如何運作」的內容
   - 應可獨立閱讀而不影響章節的敘事主線

4. **「實踐應用」**（結尾段落）
   - 恰好 5 個從本章萃取的可轉移模式
   - 每個模式：名稱 → 解決什麼問題 → 如何調適 → 需注意的陷阱
   - 具體到足以付諸行動，抽象到足以轉移至其他系統
   - 各章之間稍微變化格式以避免單調

### 語氣與風格

- **專家同儕**：像一位資深工程師為同事進行深度技術審查。不是學術的，
  不是教學的，不是行銷的。
- **直接且有主見**：「這很巧妙，因為……」/「這是錯誤的抽象，因為……」/
  「這之所以存在是因為……」
- **不灌水**：每個句子都在教某些東西，或為下一個要教的東西做鋪墊。
  如果一個句子無法證明其存在價值，就刪掉它。
- **展示取捨**：不要只描述建構了什麼。解釋什麼沒有被建構以及為什麼。
  未走之路往往更具啟發性。

### 程式碼區塊

- **僅限虛擬碼**：永遠不要原封不動地複製原始碼。展示的是模式，不是實作。
- **每章最多 3-5 個區塊**：每個區塊應為 5-15 行。
- **使用不同的變數名稱**：使用能說明概念的通用名稱，而非原始碼中的確切識別字。
- **標注為示意用途**：加入如 `// 虛擬碼——說明此模式` 或
  `// 為清晰起見做了簡化` 的註解。
- **前置上下文**：在區塊前用一句話說明它展示什麼。在區塊後用一段話說明為什麼
  這個模式重要。

### 圖表

- **Mermaid 格式**：使用 ```mermaid 圍欄程式碼區塊。這些在 GitHub 和大多數
  網頁框架中都能原生渲染。
- **每個架構概念都要有圖表**：資料流、狀態機、決策樹、時序圖、元件關係圖。
- **使用的圖表類型**：
  - `graph TD` / `graph LR` 用於架構和資料流
  - `sequenceDiagram` 用於請求/回應流程和生命週期
  - `stateDiagram-v2` 用於狀態機
  - `flowchart TD` 用於決策樹和管線
  - `gantt` 用於時序圖和平行執行
- **每章 2-4 張圖表**：複雜章節（核心迴圈、工具）多一些，聚焦型章節少一些。

### 交叉引用

- 每一章的開頭都要明確回溯引用前一章
- 當某概念將在後面展開時使用前向引用：「第 N 章將深入探討此主題」
- 每個概念只有一個正式歸屬——其他章節引用，而非重新解釋

### 一致性檢查

- 各章之間不重複修辭用語
- 標準化的「實踐應用」格式（每章 5 個模式）
- 不寫確切的檔案數量或版本號（會過時）
- 全書術語一致

## 第五階段：編輯審查

啟動 2-3 個審查代理，各覆蓋書籍的一個段落。每個審查者評估：

1. **開場品質**：是否吸引人？是否與前一章連結？
2. **流暢度**：是否有拖沓、重複、或僅列舉事實而未建構洞見的段落？
3. **內容刪減**：哪些是不服務於敘事的參考手冊式內容？哪些程式碼區塊太長？
4. **缺漏內容**：讀者會感到困惑的缺口？缺少的過渡段落？
5. **需要的圖表**：指出哪些地方可以用圖表取代大段文字。詳細描述每張圖表。
6. **跨章一致性**：語氣、格式、術語、矛盾之處。
7. **具體修改**：5-10 個需要重寫的句子/段落，附上理由。

將所有審查回饋彙整為一份排序過的行動計畫。

## 第六階段：修訂

一次性套用所有審查回饋：

- **結構調整**：拆分/合併章節、修復斷裂的引用、必要時補上遺漏的結尾章節
- **去重**：每個概念只解釋一次，其他地方使用交叉引用
- **內容刪減**：移除冗長列舉（保留模式）、精簡膨脹的段落、
  將參考資料性質的內容壓縮成表格
- **內容增補**：實作範例、真實世界的鉤子範例、在標記位置加入圖表
- **一致性**：標準化「實踐應用」段落、修復重複用語、驗證交叉引用

## 第七階段：原始碼審計

出版前，將每個程式碼區塊與原始原始碼進行比對審計：

- **替換**任何逐字或近逐字複製的區塊，改用不同變數名稱的虛擬碼
- **標注**型別簽名，加上「// 示意用途」註解
- **驗證**沒有殘留專有提示詞文字、內部常數或確切的函式實作

本書教的是模式和架構。它不應讓人得以重建確切的原始碼。
```

## 如何使用的

此提示詞產出了 **《Claude Code from Source》**——18 章分析 Anthropic AI 編碼代理架構的書籍，透過 npm 洩漏的原始碼映射（Source Map）進行逆向工程。

**數據概覽：**
- 36 個 AI 代理，橫跨 7 個階段
- 總製作時間約 6 小時
- 第一階段：6 個探索代理閱讀了 1,884 個原始碼檔案
- 第二至三階段：組織為 7 個部分、18 個章節
- 第四階段：15 個撰寫代理產出 10,320 行
- 第五階段：3 個審查代理產出 900 行回饋
- 第六階段：3 個修訂代理刪減 38%（-3,934 行），新增 25+ 張 Mermaid 圖表
- 第七階段：1 個審計代理識別出 35 處完全複製，1 個清理代理將全部替換為虛擬碼
- 最終成品：6,271 行 / 約 400 頁等量內容
